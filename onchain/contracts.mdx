## Access control

whistle uses OpenZeppelin `AccessControl` rather than a single owner address. Each sensitive action is gated by a named role:

| Role | Granted to | Guards |
|---|---|---|
| `REGISTRAR_ROLE` | Deployment admin | Adding new agents to AgentRegistry |
| `MINTER_ROLE` | Emma the Scout (agent session key) | Minting MomentNFTs |
| `MANAGER_ROLE` | Tom the Manager (agent session key) | Finalizing fantasy rosters in FantasyEntry |
| Agent session key | Each agent's hot key | Opening and settling positions in PositionManager |

No backend server holds a user's signing key. Every user-facing action (approve, allocate, withdraw) is signed in the browser.

## SettlementOracle: 2-of-3 and the challenge window

Match results are not written instantly. The flow is:

1. Any two of the three oracle signers post the same result — the 2-of-3 threshold is met.
2. A challenge window opens. During this period the result can be disputed.
3. Once the window closes without a successful challenge, the result reaches finality and PositionManager settlements can proceed.

This design prevents a single compromised signer from corrupting results and gives time to catch errors before funds move.

## PositionManager: per-match ceiling

PositionManager enforces a ceiling on how much capital can be deployed against any single match. This bounds the worst-case loss for a user whose agent acts aggressively. The ceiling is checked at position-open time; the agent session key cannot exceed it regardless of how much total capital is allocated.

## FantasyEntry: nation and jersey only

FantasyEntry stores player selections as a `(nation, jerseyNumber)` tuple. Real player names are never written to the chain. This keeps the contract lightweight and sidesteps name-rights concerns.
